# 丽滋高端医疗 UI 设计规范

说明：

- 本文用于快速查阅丽滋卡尔 web 端品牌基础规范、设计 Token、组件样式规范、业务模块规则和交付约束。

## 1. 风格定位

### 1.1 品牌定位

- 关键词：统一、简洁、易用、可扩展、一致性。
- 风格 DNA：`Premium Clinical`，即“静奢医疗科技”。
- 核心气质：
  - `Clinical Trust`：专业可信。
  - `Calm Space`：冷静洁净。
  - `Service Warmth`：服务温度。

### 1.2 视觉目标

- 整体视觉方向：
  - 白底卡片
  - 轻描边
  - 低饱和色层级
  - 克制的 VIP 点缀色
- 页面目标：高端医疗后台应体现秩序感、可信感和高信息密度下的可读性，而不是营销化、娱乐化或装饰化。

### 1.3 风格结论

- 这套规范服务的是高端医疗后台，而不是营销页、内容媒体页或娱乐化产品页。
- 核心不是视觉冲击，而是：
  - 稳定秩序
  - 高可信度
  - 高信息密度下的可读性
  - 克制但明确的交互反馈

## 2. 核心视觉语言

## 2.1 色彩系统

### 品牌与核心色

- `Navy 900` `#16213D`
  - 用途：导航、一级标题、品牌根锚点。
- `Blue 600` `#1F7AE0`
  - 用途：主按钮、激活态、关键数据。
- `Teal 500` `#18B8C9`
  - 用途：医疗服务、健康场景、辅助强调。
- `Sand 400` `#D8C29D`
  - 用途：会员、权益、等级、温和高端点缀。

### 中性与背景色

- `Ice 0` `#FFFFFF`
  - 用途：白底卡片、弹窗、表单主容器。
- `Mist 50` `#F6F9FC`
  - 用途：页面背景、次级分区、轻容器。

### 功能语义色

- `Success`：成功、完成、通过。
- `Warning`：待确认、风险提醒、即将到期。
- `Error`：错误、驳回、危险行为。
- `Info`：系统提示、说明、链接信息。
- `Neutral`：次要文本、辅助信息。

### 用色规则

- 页面禁止自由新增未命名颜色。
- 功能色必须通过语义层使用，不直接由页面自行发挥。
- 图表色需要单独语义映射，不能混入主业务色体系。
- 避免紫色、荧光绿、亮黄等与主视觉不一致的高刺激颜色进入业务主界面。

## 2.2 字体与排版

### 字体家族

- 中文目标字体：`MiSans`。
- 英文与数字字体：`MiSans`。

### 字级层级

| 层级 | Token | 字号 / 行高 | 字重 | 使用场景 |
|---|---|---:|---|---|
| 页面标题 | `type-page-title` | 42 / 52 | Bold | 页面标题、大区块主题 |
| 模块标题 | `type-section-title` | 24 / 30 | Bold | 模块区块、卡片组标题 |
| 卡片标题 | `type-card-title` | 16 / 22 | Bold | 卡片标题、分组标题 |
| 正文 | `type-body` | 14 / 22 | Regular | 正文、表单说明、信息描述 |
| 辅助说明 | `type-caption` | 12 / 18 | Regular | 注释、辅助说明、次要信息 |

### 排版规则

- 标题稳，正文轻，数字更利落。
- 正文最小可读建议为 `12 / 18`，主内容建议 `14 / 22`。
- 金额、比率、关键指标使用更强的数字视觉权重。
- 重要信息尽量用“标题 + 状态 + 动作”的三段式组织，而不是长段说明。

## 2.3 栅格、间距、圆角与阴影

### 栅格与布局

- 桌面端推荐 `12 栅格`。
- 模块外边距：`24`。
- 模块间距：`16 / 20 / 24`。
- 表单内部优先使用 `12 / 16` 节奏，避免过松或过密。

### Spacing Token

- `space-4`
- `space-8`
- `space-12`
- `space-16`
- `space-20`
- `space-24`
- `space-32`
- `space-40`

### Radius Token

- `radius-8`
- `radius-12`
- `radius-16`
- `radius-24`

### 使用建议

- 按钮、输入框优先：`12`
- 卡片优先：`16`
- 大容器优先：`24`

### 阴影规则

- 优先使用弱阴影，不依赖厚重投影制造层级。
- 阴影层级以“轻卡片 / 悬浮卡 / 弹层”为主。
- 视觉重点靠信息层级、留白、描边和状态色，而不是靠重阴影。

## 3. 页面结构规范

## 3.1 整体布局

- 统一桌面端骨架：
  - 品牌区 / 顶栏
  - 左侧导航
  - 内容区
- 左侧导航固定。
- 内容区统一 `24` 留白。
- 顶部工具栏统一样式。

### 页面装配原则

1. 页面先搭壳，再放模块。
2. 不要直接从散落组件开始拼。
3. 搜索筛选、表格、底部操作栏形成固定组合。
4. 统计卡优先表达核心数字，其次再展示趋势或标签。

## 3.2 搜索筛选区与业务流

- 搜索筛选区是后台使用频次最高的模块之一。
- 强调：
  - 结构稳定
  - 字段节奏清晰
  - 操作优先级明确
- 右侧动作建议不超过 `2` 个，如导出、批量标记。

### 筛选区规则

- 搜索与筛选栏字段节奏稳定。
- 搜索、筛选、表格、底部操作栏要形成固定组合习惯。

## 3.3 表单与页面模块组合

### 表单模块

- 按信息分组组织。
- 底部操作栏固定。
- 支持草稿自动保存。
- 复杂表单按信息块分组。
- 单个信息块字段不超过 `6` 个。
- 重点信息前置。

### 表格模块

- 由工具栏、表头、数据区、批量操作栏构成清晰纵向节奏。
- 支持搜索、导出、状态筛选、批量分组等高频操作。

### Dashboard 模块

- 模块类型：
  - 统计卡
  - 列表块
  - 图表块
- 原则：
  - 页面先搭壳，再放模块
  - 搜索筛选、表格、底部操作栏形成固定组合
  - 统计卡优先表达核心数字

## 4. 组件风格规范

## 4.1 按钮

### 类型

- `Filled`：主按钮，用于高优先级主操作。
- `Line / Ghost`：次级操作。
- `Text`：低打扰操作、查看详情、工具栏轻操作。

### 尺寸

- `Large`
- `Default`
- `Small`

### 规范

- 一个区域只保留一个主按钮。
- 列表工具栏以次级按钮和文字按钮为主。
- 主按钮优先使用蓝色，不在同一视图中叠加多种强调色。
- 视觉应“克制但明确”，不能形成过度喧闹。

### 状态

- `Default`
- `Hover`
- `Focus`
- `Disabled`

## 4.2 表单控件

### 覆盖范围

- Input
- Select
- Search
- Date Picker
- Checkbox
- Radio
- Switch
- Textarea / 备注输入

### 规范

- 强调洁净、识别度和稳定反馈。
- 依赖标签、留白和聚焦状态，不依赖重装饰。
- 表单标签清晰可读，支持两行以内输入说明。
- 错误提示优先贴近字段。
- 表单问题优先用 `Inline Alert` 靠近问题源。
- 日期与时间选择优先组合 `Date Picker + Time Picker`，优于复杂一体面板。

## 4.3 状态标签 / Badge

### 常见状态

- 已完成
- 处理中
- 待确认
- 异常
- 未分组
- 高净值

### 规范

- 状态颜色必须稳定映射。
- `Tag` 用于筛选或轻标签。
- `Badge` 用于数量提醒。
- 不建议让 Tag / Badge 同时承担强语义状态。

## 4.4 反馈与 Overlay

### 类型

- `Toast`
- `Inline Alert`
- `Alert`
- `Confirm Dialog`
- `Modal`
- `Drawer`

### 决策原则

- 能就地解决，就不要弹窗。
- 能用轻提示完成，就不要升级为强打断。
- 反馈越重，打断越强。

### 使用边界

- `Toast`
  - 轻成功、系统提示
  - 停留时间建议 `2s` 自动消失
- `Inline Alert`
  - 表单字段错误、就地说明
- `Confirm Dialog`
  - 仅用于不可逆或高风险操作
- `Modal`
  - 强确认、短任务
- `Drawer`
  - 长表单、侧边编辑、上下文补充
  - 出现后主页面仍应可辨识

### 文案原则

- 错误信息必须说明原因和修复动作。
- `Warning` 只用于风险提醒。
- `Success` 不应大面积占位。

## 4.5 导航

### 组件

- Sidebar
- Breadcrumb
- Tabs
- Filter Strip
- Pagination

### 规则

- 左侧导航负责全局层级。
- Tabs 负责局部切换。
- Breadcrumb 只用于说明路径。
- 导航不追求强视觉，而追求稳定秩序与快速定位。

### 分页

- 页码型分页用于长列表。
- 与筛选、批量操作栏组合使用。
- 切换动画属于中等级别动态，可用 `180ms` 左右节奏。

## 4.6 数据展示

### 表格 Table

- 强调“结构先于花哨”。
- 应具有：
  - 工具栏
  - 表头
  - 数据区
  - 批量操作栏
- 行高应舒展，边线要弱。
- 重点信息通过字重和状态色表达。

### 统计卡 Stat Card

- 优先表达核心数字。
- 趋势、标签、说明放在次级。
- 看板模块遵循“强信息、弱装饰”。

### 指标条 Metric Bars

- 适合展示会员贡献、复诊转化等比例指标。
- 使用语义色和数字权重表达优先级。

## 4.7 卡片

### 类型

- 客户摘要卡
- Dashboard 卡
- 服务卡
- Rich Card

### 样式规则

- 白底卡片 + 轻描边为默认。
- 通过层级、留白和轻量操作建立高端感。
- 避免厚重装饰。
- 深色底看板卡可用于强调型业务块，但应搭配单一重点色，不能花哨。

### 内容组织

- 标题
- 状态 / 标签
- 核心数字或关键信息
- 次级说明
- 轻量操作

## 4.8 图标

### 规则

- 以线性图标为主，面性慎用。
- 常用尺寸：`16 / 20 / 24 / 32`
- 线宽建议：`1.5 - 2`

### 使用建议

- 导航图标统一 `20`
- 按钮附属图标统一 `16`
- 业务卡片与数据模块优先线性图标
- 避免彩色插画式图标

### 适用位置

- 按钮内图标
- 标签图标
- 导航 / 表格工具栏
- 空状态 / 功能说明
- 仪表盘强调图标

## 4.9 Utility 组件

### Divider

- 仅用于建立轻层级。
- 颜色始终弱于卡片边线。
- 不承担主视觉分割。

### Slider

- 用于范围和程度选择。

### Rate

- 仅建议用于评价结果展示。
- 不建议在主业务流程中频繁出现。

## 4.10 Loading / Skeleton / Empty

### Loading

- 超过 `400ms` 展示 loading。
- 超过 `2s` 必须给出原因或进度说明。

### Skeleton

- 优先保留真实版式轮廓。
- 避免列表突然跳高。

### Empty

- 必须同时给出“原因 + 下一步动作”。
- 高价值页面应提供重置筛选或返回上一步。

## 4.11 微布局模块

### Steps / Progress

- 适合线性流程。
- 最多 `4 - 5` 步。
- 超过后改用分页或分组导航。

### Collapse / Accordion

- 仅在需要展开更多解释、明细或说明时使用。
- 不建议把大量核心信息都塞进折叠区。

### List / Grid / Card Cell

- 列表可分为普通列表、带图标列表、带操作列表。
- 网格模块使用统一卡片宽度。
- 不依赖任意拉伸形成层级。

## 4.12 时间与上传类组件

### Popover / Tooltip

- Popover 用于轻决策和少量动作。
- Tooltip 只承载一行补充说明。

### Upload

- 必须说明：
  - 支持格式
  - 数量限制
  - 当前进度
  - 失败恢复方式
- 医疗资料上传尤其要强调状态透明。

### Calendar / Time Picker

- 日历默认展示月视图。
- 保留今天态与选中态。
- 时间粒度建议 `15 分钟`。
- 医疗预约场景优先展示空闲时间，而不是让用户盲选。

## 5. Design Token 规范

## 5.1 Token 分层

- `Brand Token`
  - 如 `navy` / `blue` / `teal` / `sand`
- `Semantic Token`
  - 如 `primary` / `success` / `warning` / `error` / `info`
- `Component Token`
  - 进入按钮、表单、卡片、表格、反馈组件等

## 5.2 推荐命名

### 颜色

- `$color-brand-navy-900`
- `$color-primary-600`
- `$color-secondary-500`
- `$color-accent-sand-400`
- `$color-surface-0`
- `$color-bg-mist-50`

### 状态

- `$state-success-500`
- `$state-warning-500`
- `$state-error-500`
- `$state-info-500`
- `$semantic-neutral`

### 类型

- `type-page-title`
- `type-section-title`
- `type-card-title`
- `type-body`
- `type-caption`

## 5.3 Token 使用规则

1. 品牌色先定义 Brand Token。
2. 页面状态统一走 Semantic Token。
3. 组件交互统一走 State Token。
4. 页面和业务模块禁止新增未命名颜色。
5. 建议在 Figma 同步为变量，在前端映射为 JSON / CSS Variables。

## 6. 交互与可访问性

## 6.1 交互原则

- 可预期：同一动作在不同页面保持相同行为。
- 轻反馈：成功和错误都走短路径反馈。
- 高确认：破坏性操作必须二次确认。
- 低打扰：提示服务任务，而不是打断任务。

### 标准任务闭环

- 输入
- 校验
- 提交
- 反馈
- 完成

## 6.2 组件状态

- Default
- Hover
- Focus
- Disabled

所有输入控件、按钮和反馈类组件都必须遵守统一状态映射。

## 6.3 动效节奏

- `120ms`
  - 按钮 hover、小状态切换
- `180ms`
  - 抽屉、筛选展开、分页切换
- `240ms`
  - Modal、页面骨架进入

### 动效原则

- 短、轻、少。
- 帮助理解结构变化和状态切换。
- 避免娱乐化表达。

## 6.4 可读性与无障碍

- 正文最小建议：`12 / 18`
- 主内容建议：`14 / 22`
- 对比度建议：`>= 4.5:1`
- 点击热区：`>= 40px`

### 键盘规则

1. 弹窗与抽屉必须支持 `Esc` 退出。
2. Focus 态颜色或描边必须清晰可见。
3. Tab 顺序可见，Enter 可提交。

### 错误提示规则

- 字段错误优先贴近字段。
- 跨字段错误放在分组顶部。
- 提交后顶部汇总只能作为补充，不能替代就地反馈。

## 7. 文案与交付

## 7.1 文案规则

- 错误文案必须说明原因与解决方向。
- 反馈文案应克制、专业、可执行。
- 重要信息尽量使用“标题 + 状态 + 动作”的组织方式。

## 7.2 命名规则

推荐格式：`模块 / 组件 / 状态 / 尺寸`

示例：

- `Button / Primary / Default`
- `Table / Toolbar / Batch`
- `Modal / Confirm / Delete`

## 7.3 标注规范

设计稿标注应包含：

- 尺寸 / 间距
- Token 名称
- 交互状态
- 异常说明

开发备注可包含：

- 表头固定
- 底部操作栏滚动后仍可见
- 日期选择默认今日
- Toast 停留 2s 自动消失

## 7.4 交付 Checklist

- 颜色、间距、圆角已映射为 Token。
- 字号层级与实际页面一致。
- 状态色命名统一。
- 组件覆盖默认 / hover / focus / disabled。
- 表单、表格、弹窗均有业务示例。
- 标注了关键组件的使用边界与禁忌。
- 页面定义了结构壳与模块拼装关系。
- 重点交互路径和异常态已说明。
- 标题、按钮、状态可被直接理解，不依赖猜测。
- 明确字体替代情况。
- 输出了命名规范与开发备注。

## 8. 适用与禁忌

## 8.1 适用范围

- 高端医疗后台
- 客户档案与客户中心
- 医疗预约与排班管理
- 服务方案与会员权益管理
- 审阅、回访、运营、数据管理类页面

## 8.2 不建议偏离的方向

- 不应做成营销化页面。
- 不应做成娱乐化、内容媒体化页面。
- 不应靠重阴影、强装饰和高刺激颜色制造层级。
- 不应让业务页面自由新增颜色或破坏状态映射。

## 8.3 高级业务模块边界

### Tree Module

- 左侧只负责结构定位。
- 右侧负责详情和批量操作。
- 树结构只表达层级，不表达复杂状态。

### Bottom Action Bar

- 适用于长表格和多选场景。
- 固定位置，不让用户滚动到底部才找到操作。

### Carousel

- 仅用于少量卡片切换。
- 不应成为核心导航。
- 后台场景默认关闭自动轮播。
- 卡片数量建议控制在 `3 - 5` 个。

## 9. 适合前端实现的落地要点

1. 先建 Token，再建组件，再建页面模板。
2. 按“页面壳层 → 筛选区 → 表格 / 表单 → 反馈 → 高级模块”顺序实现。
3. 主界面统一白底卡片与轻描边，避免组件各自定义视觉语言。
4. 把状态系统做成全局语义映射，不让业务自行染色。
5. 表格、筛选、底部操作栏、抽屉、Modal、Toast、Empty、Skeleton 应作为首批标准化组件。
6. 医疗预约、档案、排班、审阅等页面优先提供真实业务模板，而不是只交组件碎片。
